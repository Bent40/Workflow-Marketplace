# FreshBox alignment situation (consultant -> client)

A freelance product consultant is proposing to build the FreshBox web app for a client (the chef-owner). They need stakeholder alignment BEFORE starting, so the consultant is preparing an alignment document to share with the owner.

## Raw inputs the consultant has gathered
- **Realization:** The owner currently runs FreshBox via phone/email and a basic landing page. They lose subscribers because "how it works" and "can I skip a week" are unclear, and the owner spends hours answering the same emails. Evidence: ~40 repetitive support emails/week; 22% of churned customers cite confusion about skipping.
- **Who we help:** Busy two-income parents who want predictable weekly dinners and the safety of penalty-free skip/pause. Helping them helps the owner retain subscribers.
- **How we see it working:** A clear chef-led web app that explains the weekly cycle up front, makes skip/pause obvious, and lets parents subscribe and self-manage.
- **Paths explored:** (a) patch the existing landing page, (b) adopt an off-the-shelf subscription template, (c) build a focused custom web app via the WDS design-first process.
- **Recommended:** Path (c) — a focused custom build, because the differentiator (chef-led, easy-skip) needs bespoke messaging the templates can't carry.
- **Value if we do it:** higher 8-week retention, fewer support emails, a credible brand presence. Success measured by retention % and weekly support-email count.
- **Cost of inaction:** continued churn, owner time drained on email, competitors capturing the local market.
- **Commitment:** ~6 weeks of design+build, small budget, owner available for weekly reviews. Risk: owner's limited time for feedback.

The consultant wants an alignment document (a pitch) covering all of this so the owner can say yes before work starts.
