- **Origin**: user-approved roadmap skill ("go ahead with the roadmap skills").
  Named as a governance-approved-but-unbuilt roadmap item in the 3D pack charter
  from the prior build; this session builds it.
- **Governance**: CREATE-NEW, already greenlit in the charter roadmap. Distinct
  from the other pack skills: `3d-create` runs the pipeline, `3d-help` routes,
  `3d-troubleshoot` diagnoses a *failure* — this one gets a bare box **ready**
  (base software + model downloads + smoke test). No overlap.
- **Name**: `3d-setup` (the pack's `3d-` prefix; matches the roadmap name the
  charter and `3d-help` already advertise).
- **Classification**: Simple Workflow, fully inline (~80 lines). Reads
  `models.md` + `hardware-scouting.md` for the exact repos/paths/sizes rather
  than duplicating them.
- **Content bar — what an LLM would get wrong unprompted**: the **no-pip-into-
  the-venv** hard constraint (a "fix" that edits the venv is the obvious wrong
  move); **right-sizing** the download set to the goal (don't pull ~30 GB for a
  print-only user); **verify-in-order** (ready == a real smoke test yields a
  WATERTIGHT STL, not "files exist"); wrong-path-reads-as-missing.
- **Trigger design**: "3d setup", "set up the 3D box", "which models do I need",
  "get my machine ready for 3D", "pipeline can't find its models". Negatives:
  database-schema setup, Node/React project setup, and pip-install-into-an-ML-env
  must not fire it.
- **Escort behavior**: one step at a time, confirm downloads, verify before
  declaring ready — the pack's escort model applied to setup.
- **Validation mode**: bmad-eval-runner unavailable in-container (no Docker/API
  key) → degraded in-session validation (fresh validator subagent: trigger
  proxy + artifact evals graded vs `expectations`). Re-run hermetically when
  Docker + key are available.
- **Validation result (2026-07-07, degraded)**: trigger sanity **8/8 match**,
