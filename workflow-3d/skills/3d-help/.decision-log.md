# 3d-help — decision log

## Session 2026-07-07 (branch claude/3d-pack-integration-eutnkd)

- **Origin**: user-requested, same effort as `3d-mode` — the "guide anyone
  installing it into creating anything they want" requirement lands squarely
  here. This is the 3D pack's `bmad-help` / `writer-help` analogue.
- **Governance**: CREATE-NEW under the same 3D-pack verdict. The guide is
  distinct from `3d-mode` (posture loader) and `3d-create` (does the
  work): it reads box + input state, asks the decisive output question, and
  routes. Not covered by `bmad-help` (BMAD/dev lifecycle).
- **Name**: `3d-help` (parallels `writer-help`).
- **Classification**: Simple Workflow, fully inline (~75 lines).
- **Content bar**: the decisive move is the **output goal question**
  (print/game/minecraft → engine), because that single decision routes the whole
  pipeline. Probe is inventory-only (readiness + inputs), never loads/uploads
  art, never runs a heavy generation. Honest-roadmap rule carried verbatim from
  writer-help: never recommend the unbuilt `3d-setup`/`3d-troubleshoot`
  — name the docs substitute. Watertight-honesty carried even into "is it
  printable?" answers.
- **Trigger design**: "3d help", "what can I make", "where do I start",
  "is my box ready", plus direct which-engine questions. Negative boundary:
  `bmad help` and sprint/README help must not fire it.
- **Validation mode**: degraded in-session (as 3d-mode); re-run with
  bmad-eval-runner when Docker + key are available.
- **Validation result (2026-07-07, degraded)**: trigger sanity **8/8 match**,
  artifact evals **4/4 PASS** via a fresh validator subagent. Polish applied per
  the validator: the Minecraft bullet now states the "voxelization discards fine
  geometry → cheapest engine" rationale explicitly (was only inferable).
